PART 1 · 总论

技能数据分布规范

消除 DataTable 与 GAS 的双源冲突，建立四层单一权威数据源（SSOT）。通过多层默认配置体系将策划创建一个新技能的初版时间从 20-30 分钟缩短到 3-5 分钟。

四层架构

每条技能数据只回答三个问题：谁写、谁读、何时写。四层按职责严格隔离，通过 ID/路径引用关联。

L0 技能注册表 纯索引层 — ID、名称、图标、分类、解锁条件 注册 · 索引 · UI
L1 GAS Ability 唯一运行时权威源 — 距离、区域、状态、权限、阶段 逻辑 · 权威 · SSOT
L2 Timeline 纯时序表现 — 动画、特效、音效、碰撞帧、阶段标记 表现 · 时间 · 标记
L3 FlowCanvas 效果分发中心 — 命中后伤害、Buff、受击反馈逻辑 效果 · 事件 · 分支

核心设计原则

⊕
单一权威源
每个字段有且只有一个写入点。GAS 为运行时唯一权威。DataTable 收窄为纯注册索引。
◈
多层默认继承
代码约定 → 武器默认 → 类别模板 → 实例覆盖。策划只需配置"与默认不同的"字段。
⬡
模板驱动创建
15 类技能模板 + 6 种 Timeline 模板 + 5 种 Flow 模板 + 7 种 GE 模板，覆盖 95% 场景。
⟐
约定优于配置
资产路径由命名约定自动推导。~90% 路径字段免填，消除拼写错误和不一致。
◉
命中反馈统一
Hit Feel Profile 归拢所有打击感参数。顿帧、震屏、受击反应、击退一站式配置。
⧉
变体优于复制
相似技能通过变体继承只存差异字段。强化版、NPC版技能创建从 20 分钟降到 2 分钟。

L0

L2

Timeline — "这个技能看起来怎样"

负责"第几帧播什么表现"和"第几帧触发什么阶段信号"，不包含任何战斗逻辑。

✓ 属于 Timeline
  • 动画播放/切换时机
  • 特效/音效触发时机
  • 碰撞帧/伤害帧时机
  • 投射物发射时机
  • 位移曲线、相机表现
  • 阶段标记（Phase Markers）
✗ 不属于 Timeline
  • 各阶段允许什么操作 → L1 Phase GE
  • GAS 效果应用 → L3 FlowCanvas
  • 施法条件/消耗/冷却 → L1 GAS
  • 命中后效果逻辑 → L3 FlowCanvas

阶段标记（Phase Markers）可视化

策划在 Timeline Editor 中拖拽标记位置，与动画关键帧对齐。标记本身只携带 phase_id，不含任何逻辑参数。

阶段模式：
运行时信号接口

PhaseMarker 通过事件总线（而非直接引用）与 L1 通信，确保 L2 → L1 单向依赖。

L2 端 · TimeAction_PhaseMarker
OnEnter() {
  SpellContext.RaisePhaseEvent({
    PhaseId:    phase_id,
    SpellId:    SpellContext.SpellTid,
    Normalized: Timeline.NormalizedTime
  });
}
L1 端 · GA PhaseMarker 监听
OnPhaseEvent(e) {
  GE = PhaseEffectMap[e.PhaseId]
  ASC.ApplyGE(GE)
    → 自动 RemoveEffectsWithTag: Phase.*
    → 授予新 Tags (Permission.*, State.*)
  Broadcast(OnPhaseChanged)
}
关键约束：PhaseMarker 只能发信号，不能直接调用 GAS API。L2 不持有 L1 引用，替换 GAS 框架时 Timeline 无需改动。
L3

只负责"命中/触发之后执行什么效果逻辑"。纯逻辑流图，事件驱动。

✓ 属于 FlowCanvas

  • GE 应用（伤害/Buff）
  • Tag 操作（命中触发）
  • 条件分支（血量/Tag/堆叠）
  • 受击反馈编排
  • 投射物命中处理
✗ 不属于 FlowCanvas
  • 施法者阶段状态管理 → L1 Phase GE
  • 动画/特效播放时机 → L2 Timeline
  • 阶段转换/操作权限 → L1 Phase GE

将技能时间帧的"什么时候"（L2 标记）和"允许什么"（L1 规则）分属两层，通过 GE + Tag 融入 GAS 体系。

ℹ 连招类技能在 Active 后额外插入 Cancelable 阶段用于接招，之后才进入后摇。

取消窗口 vs 后摇 —— 关键对比

这是策划最容易混淆的两个阶段。核心区别：取消窗口 = 可以接招但不能走路；后摇 = 可以走路但不能接招。

PART 4 · 韧性击破

Break State 击破状态

韧性归零不仅是一次受击反应，更是一个持续数秒的脆弱窗口期。Break 构成"积攒削韧 → 击破 → 奖励窗口 → 恢复"的战斗节奏循环，用 Duration GE + Tag 原生实现。

受击反应 vs Break 状态

受击反应 (Hit Reaction)

一击的即时表现：硬直、击退、击飞。持续 ~0.1-0.5s。
每次命中都可能触发，由 HitFeelProfile 和韧性计算决定。

Flinch · Stagger · Knockback · Knockdown · Launch
Break 状态 (Break State)

韧性归零后的持续脆弱窗口：3-5 秒。
期间受伤增幅、特殊交互开放（处决/弱点攻击）、目标无法/有限行动。

State.Broken + DamageTaken ×1.3~2.0

EXPLORE · AI + GAS 工作流

AI 协作工作流探索

探索面向 AI+人类协作的三层架构 —— 意图层 → 配置层 → 运行时层，GAS 不替换，只加创作入口。

GAS 的 AI 摩擦点

当前 GAS 四层架构在 AI 协作场景下的核心阻力。

★★★ 资产碎片化
创建 1 个普攻段需触碰 8 个实体（GA + GE + HFP + Phase GE×4 + TL + Flow + Route），分布在 4 层、4 种格式、3 种编辑器中。
现状
AI 需要理解并协调 8 个实体的引用关系，任何一条链断裂→ Error
意图层方案
一技能一 YAML 文件，自包含所有信息，AI 只需操作单一文本
★★★ 隐式上下文
Phase GE 选择、HFP 选择、inputWindow 对齐、继承链回退 —— 这些"部落知识"对 AI 是缺失上下文。
现状
Phase GE 有 15 个变体，选择依赖经验而非规则；0/null = fallback 语义不直观
意图层方案
语义标签 + 显式继承：mode: mobile → 自动映射 Phase GE
★★ 二进制格式
Unity .asset 使用 GUID 引用 + 二进制序列化，AI 无法纯文本读写。
现状
每次编辑需 AI → Bridge → Unity API → 返回结果，链路长延迟高
意图层方案
纯 YAML 文本，AI 直接读写，最终由 Compiler 编译到 .asset
★★ 校验滞后
当前校验为离线批量模式，AI 无法在同一轮对话内完成 生成→校验→修正 闭环。
现状
生成→写入→触发CI→读报告→修正… 闭环 >30s
意图层方案
Schema 内嵌约束 + 同步校验引擎，Resolve 即校验，闭环 <2s

结论

  

    
GAS 仍是最优运行时，问题在创作层

    

      
GAS 的 Tag 驱动权限、GE 统一效果容器、ASC 生命周期管理和网络同步支持在 ARPG 场景下无可替代。

      
但其数据模型以引擎运行时为中心，与 AI 的概念驱动思维存在表示层错配。解决方案不是替换 GAS，而是在其上增加意图层作为 AI 和策划的统一创作入口，通过 Resolver + Compiler 自动编译到现有 GAS 资产体系。

      
预估效率提升：同等技能创建从 ~150min 降至 ~42min（节省 72%），主要节省在 GA/GE 手动填写和 FlowCanvas 搭建环节。
